The Log-Strctured Merged-Bush


Key-value stores (KV-stores) are used as the storage back-bone across numerous modern applications to rapidly ingest and persist data. There exist two design paradigms for such write-optimized KV stores, each with distinct properties. (1) The log-structured hash table (LSH-table) logs entries in stor- age and uses an in-memory hash table to map their keys.(2) The log-structured merge-tree (LSM-tree) sort-merges data across multiple levels in storage and uses in-memory Bloom filters and fence pointers to search the levels. LSH-table offers cheap writes but suffers from slow range lookups and requires a lot of memory. On the other hand, LSM-tree performs well on range lookups and requires less memory but writes are much more expensive. In this way, modern KV-stores have to compromise by choosing one or the other. In this work, we alleviate this compromise by introducing the Log-Structured Merge-Bush (LSM-bush), a novel data structure that enables new and even dominating trade-offs in-between LSH-table and LSM-tree that were not possible or known before.

Background


LSH-table. LSH-table buffers application writes in memory and flushes them to a log in storage whenever the buffer is full. A hash table is maintained in memory mapping from every key to the location of the corresponding entry in the log.

LSM-Tree. LSM-tree buffers application writes in memory, flushes the buffer as a sorted run to storage whenever it is full, and it merges similarly-sized runs across multiple levels of exponentially increasing capacities. Modern LSM-tree designs use in-memory fence pointers to allow lookups to quickly find the relevant key range at each run, and they use in-memory Bloom filters to allow point lookups to skip accessing runs that do not contain a target entry.

Problem



The Practical Problem: Suboptimal Data Storage. Today, KV-stores must choose between mutually exclusive sets of properties: (1) fast writes with impractical range lookups and immodest memory as with LSH-table, or (2) costly writes with fast range lookups and modest memory as with LSM-tree. With applications becoming increasingly diverse, we need tailored designs to fully optimize I/O and memory. The binary choice between LSH-Table and LSM-tree, however, forces a rigid compromise one way or the other.


The Research Challenge: Bridging the Design Abyss. LSH-table and LSM-tree delineate a vast and hitherto unexplored design abyss. Today, we neither know how to navigate the space in-between nor what trade-offs would be possible to achieve along it. In fact, we do not even know whether LSH-table and LSM-tree are the best possible extreme points in the space, or whether it is possible to narrow the space by devising dominating designs with all-around better trade-offs. In this paper, we take a stab at this space and show that there are good and even dominating designs in-between.

Solution: LSM-Bush


We introduce the Log-Structured Merge-Bush (LSM-bush), a new data structure with dominating trade-offs in-between LSH-table and LSM-tree. Similarly to LSM-tree, LSM-bush buffers writes in memory, merges them as sorted runs across multiple levels in storage, and uses in-memory fence pointers and Bloom filters to facilitate lookups. The core technical innovation is setting explosively growing capacity ratios between adjacent pairs of smaller levels, while at the same time merging at smaller levels only when they fill up. As a result, newer data is merged more lazily as smaller levels gather larger numbers of runs before merging them. In this way, LSM-bush eliminates most merging relative to LSM-tree to achieve asymptotically faster writes. On the other hand, larger levels gather fewer though larger runs, meaning they get progressively more read-optimized. This allows LSM-bush to achieve fast and scalable range lookups. LSM-bush allocates its Bloom filters such that false positives at its lower, more write-optimized levels are increasingly improbable. This enables fast and scalable point lookups.
